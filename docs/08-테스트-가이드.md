## 테스트 구조 개요

프로젝트 테스트는 **3계층 + 라이브 테스트**로 구성되어 있으며, 총 **65개** 테스트가 존재한다.

| 계층 | 테스트 수 | Spring Context | 설명 |
|------|-----------|----------------|------|
| 단위 테스트 | 52개 | 불필요 | 개별 클래스의 로직 검증 |
| 통합 테스트 | 7개 | 필요 | Spring Boot 전체 컨텍스트 기반 검증 |
| E2E 테스트 | 5개 | 필요 | MCP 서버 엔드투엔드 검증 |
| 라이브 테스트 | 1개 | 필요 | 실제 외부 API 호출 (수동 실행) |

### 디렉토리 구조

```
src/test/
├── java/com/apiagent/
│   ├── config/              # 설정 관련 단위 테스트
│   ├── executor/            # DuckDB 실행기 단위 테스트
│   ├── client/rest/         # REST 클라이언트 단위 테스트
│   ├── util/                # 유틸리티 단위 테스트
│   ├── context/             # 요청 컨텍스트 단위 테스트
│   ├── agent/               # 에이전트 단위 테스트
│   ├── mcp/                 # MCP 도구 단위 테스트
│   ├── recipe/              # 레시피 단위 테스트
│   ├── integration/         # 통합 테스트
│   └── e2e/                 # E2E 테스트 + 라이브 테스트
└── resources/
    └── application-test.yml # 테스트 전용 설정
```

## 테스트 실행 방법

### 사전 준비

JDK 21이 설치되어 있어야 합니다:

```bash
java -version
# openjdk version "21.0.x" ...
```

테스트 의존성 (`build.gradle.kts`):
- `spring-boot-starter-test` - JUnit 5 + AssertJ + Mockito 포함
- `junit-platform-launcher` - JUnit 5 실행기

### 테스트 실행

```bash
# 단위 + 통합 + E2E 테스트 실행 (라이브 테스트 제외)
./gradlew test

# 라이브 테스트 포함 전체 실행
OPENAI_API_KEY=sk-... ./gradlew test -PincludeLive=true

# 상세 출력
./gradlew test --info

# 특정 테스트 클래스
./gradlew test --tests "com.apiagent.e2e.McpServerE2eTest"

# 특정 테스트 메서드
./gradlew test --tests "com.apiagent.executor.DuckDbExecutorTest.extractTables_list_usesName"

# 특정 패키지의 모든 테스트
./gradlew test --tests "com.apiagent.integration.*"

# 실패한 테스트만 재실행
./gradlew test --rerun

# 테스트 리포트 (build/reports/tests/test/index.html)
./gradlew test && open build/reports/tests/test/index.html
```

### Python 원본과의 차이

| Python | Spring Boot |
|--------|------------|
| `pytest tests/ -v` | `./gradlew test` |
| `pytest -k "recipe"` | `./gradlew test --tests "*Recipe*"` |
| `pytest --lf` | `./gradlew test --rerun` |
| `ruff check` (린터) | 별도 린터 없음 (IDE 의존) |
| `ty check` (타입 체크) | 컴파일 시 자동 (Java는 정적 타입) |
| `pytest-asyncio` | 불필요 (Virtual Threads = 동기 코드) |

## 통합 테스트 (7개)

`@SpringBootTest(webEnvironment = RANDOM_PORT)` + `@ActiveProfiles("test")`로 실제 Spring Context를 기동하여 빈 주입, 프로퍼티 바인딩, HTTP 요청을 검증한다.

### ApplicationContextTest (4개)

- `contextLoads` — Spring Context 정상 로딩 확인
- `핵심_빈이_주입된다` — McpToolProvider, AgentService, Properties 빈 존재 확인
- `프로퍼티가_정상_바인딩된다` — mcpName, serviceName, modelName 등 필수 프로퍼티 검증
- `actuator_health_응답` — `/actuator/health` 엔드포인트 200 OK + "UP" 응답

### RequestContextFilterIntegrationTest (3개)

- `필터가_헤더로_컨텍스트를_설정한다` — `X-Target-URL`, `X-API-Type`, `X-Target-Headers` 헤더를 통한 컨텍스트 설정
- `필터_후_ThreadLocal이_정리된다` — 요청 완료 후 `RequestContextHolder`의 ThreadLocal 정리 확인
- `필수_헤더_없어도_에러_발생하지_않는다` — 헤더 미전송 시에도 정상 동작

## E2E 테스트 (5개)

### McpServerE2eTest

MCP 서버의 빈 등록, 서버 정보, 도구 기능, SSE 엔드포인트, 에러 처리를 종합 검증한다.

- `MCP_서버_빈이_등록된다` — `McpSyncServer` 빈 존재 확인
- `MCP_서버_정보가_올바르다` — 서버 이름 "API Agent", 버전 "0.1.0" 확인
- `MCP_서버_도구_기능이_활성화된다` — capabilities.tools 활성화 확인
- `SSE_엔드포인트가_연결_가능하다` — `/sse` 엔드포인트 200 OK + `text/event-stream` Content-Type 확인
- `잘못된_MCP_메시지_경로에_에러_반환` — 존재하지 않는 엔드포인트에 대한 에러 응답 확인

## 라이브 테스트

### McpLiveQueryE2eTest (`@Tag("live")`)

실제 OpenAI API를 호출하여 공개 GraphQL API([Countries API](https://countries.trevorblades.com/graphql))에 자연어 질의를 수행한다.

- `공개_GraphQL_API에_자연어_질의` — "한국의 수도는?" 질의 → 응답에 "Seoul" 또는 "서울" 포함 확인

**주의사항:**
- `OPENAI_API_KEY` 환경변수가 설정되어 있어야 한다

- 기본 `./gradlew test` 실행 시 **제외**된다
- 실행: `OPENAI_API_KEY=sk-... ./gradlew test -PincludeLive=true`

## 테스트 환경 설정

### `application-test.yml`

```yaml
spring:
  main:
    allow-bean-definition-overriding: true   # 테스트용 빈 오버라이딩 허용
    allow-circular-references: true          # 순환 참조 허용 (MCP 서버 구조상 필요)
  ai:
    openai:
      api-key: ${OPENAI_API_KEY:test-dummy-key-for-context-loading}  # 더미 키로 컨텍스트 로딩
```

- 통합/E2E 테스트는 `@ActiveProfiles("test")`로 이 설정을 활성화한다
- `OPENAI_API_KEY`가 없으면 더미 키로 대체되어 컨텍스트 로딩은 성공하지만, 실제 API 호출은 실패한다

### `build.gradle.kts` — 라이브 테스트 제외 설정

```kotlin
tasks.withType<Test> {
    useJUnitPlatform {
        if (!project.hasProperty("includeLive")) {
            excludeTags("live")   // -PincludeLive 미지정 시 @Tag("live") 테스트 제외
        }
    }
    jvmArgs("--enable-preview")   // Java 21 Preview 기능 활성화
}
```

## 테스트 패턴

### 순수 단위 테스트 (주요 패턴)

이 프로젝트의 대부분의 테스트는 **순수 단위 테스트**입니다. `@SpringBootTest`나 `@DataJpaTest` 같은 슬라이스 테스트는 사용하지 않습니다.

```java
class RecipeNamingTest {

    @Test
    void sanitize_removesSpecialChars() {
        assertThat(RecipeNaming.sanitize("Get Recent Flights!"))
            .isEqualTo("get_recent_flights");
    }

    @Test
    void sanitize_nullReturnsRecipe() {
        assertThat(RecipeNaming.sanitize(null))
            .isEqualTo("recipe");
    }

    @Test
    void deduplicate_appendsSuffix() {
        Set<String> seen = new HashSet<>(Set.of("get_flights"));
        assertThat(RecipeNaming.deduplicate("get_flights", seen, 40))
            .isEqualTo("get_flights_2");
    }
}
```

### AssertJ 사용 패턴

```java
// 기본 검증
assertThat(result).isEqualTo("expected");
assertThat(result).isNotNull();
assertThat(result).isTrue();

// 컬렉션 검증
assertThat(list).hasSize(3);
assertThat(list).contains("item1", "item2");
assertThat(map).containsKey("key");

// 예외 검증
assertThatThrownBy(() -> holder.require())
    .isInstanceOf(IllegalStateException.class)
    .hasMessageContaining("not set");

// 문자열 검증
assertThat(result).startsWith("r_");
assertThat(result).hasSize(40);
```

## Mockito 모킹 패턴

### 기본 패턴: `@Mock` + `@InjectMocks`

외부 의존성이 있는 서비스를 테스트할 때:

```java
@ExtendWith(MockitoExtension.class)
class RestApiClientTest {

    @Mock
    private RestClient restClient;

    @InjectMocks
    private RestApiClient restApiClient;

    @Test
    void execute_getRequest_success() {
        // given
        when(restClient.get()
            .uri(any())
            .retrieve()
            .body(Map.class))
            .thenReturn(Map.of("id", 1, "name", "Alice"));

        // when
        Map<String, Object> result = restApiClient.execute(
            "GET", "/users/1", null, null, null,
            "http://test", Map.of(), List.of());

        // then
        assertThat(result.get("success")).isEqualTo(true);
    }
}
```

### unsafe 메서드 차단 테스트

```java
@Test
void execute_postWithoutAllowList_blocked() {
    Map<String, Object> result = restApiClient.execute(
        "POST", "/users", null, null, Map.of("name", "John"),
        "http://test", Map.of(), List.of());  // 빈 allowList

    assertThat(result.get("success")).isEqualTo(false);
    assertThat(result.get("error").toString())
        .contains("not allowed");
}

@Test
void execute_postWithMatchingPattern_allowed() {
    when(/* ... mock setup ... */).thenReturn(Map.of("id", 1));

    Map<String, Object> result = restApiClient.execute(
        "POST", "/search", null, null, Map.of("q", "test"),
        "http://test", Map.of(), List.of("/search"));  // /search 허용

    assertThat(result.get("success")).isEqualTo(true);
}
```

## DuckDB 실제 사용 패턴

`DuckDbExecutor` 테스트는 **실제 DuckDB 인스턴스**를 사용합니다. 인메모리로 동작하므로 빠르고 부작용이 없습니다.

```java
class DuckDbExecutorTest {

    private final DuckDbExecutor executor = new DuckDbExecutor();

    @Test
    void extractTables_listResponse_usesName() {
        List<Map<String, Object>> data = List.of(
            Map.of("id", 1, "name", "Alice"),
            Map.of("id", 2, "name", "Bob")
        );

        var result = executor.extractTables(data, "users");
        assertThat(result.tables()).containsKey("users");
        assertThat(result.tables().get("users")).hasSize(2);
    }

    @Test
    void executeSql_simpleQuery_success() {
        Map<String, List<?>> data = Map.of(
            "users", List.of(
                Map.of("id", 1, "name", "Alice"),
                Map.of("id", 2, "name", "Bob")
            )
        );

        Map<String, Object> result = executor.executeSql(data,
            "SELECT * FROM users WHERE id = 1");

        assertThat(result.get("success")).isEqualTo(true);
        List<?> rows = (List<?>) result.get("result");
        assertThat(rows).hasSize(1);
    }
}
```

### Python 원본과의 테스트 차이

| Python 원본 | Spring Boot |
|-------------|------------|
| `pytest` + `pytest-asyncio` | JUnit 5 + AssertJ |
| `unittest.mock.AsyncMock` | Mockito (`@Mock`) |
| `httpx.AsyncClient` 모킹 | `RestClient` 모킹 |
| `ContextVar` 테스트 초기화 | ThreadLocal 불필요 (매 테스트 격리) |
| 15 테스트 파일 | 15 테스트 파일 (단위 11 + 통합 2 + E2E 2) |
| `ruff` 린팅 | 컴파일 타임 체크 |
| GitHub Actions CI | (설정 시 동일 패턴 적용 가능) |

## 알려진 이슈

### SSE 엔드포인트 테스트에서 `HttpURLConnection` 사용

`McpServerE2eTest.SSE_엔드포인트가_연결_가능하다` 테스트는 `TestRestTemplate` 대신 `HttpURLConnection`을 직접 사용한다. SSE는 스트리밍 응답이므로 `TestRestTemplate`이 응답 완료를 무한 대기하기 때문이다. `connectTimeout`과 `readTimeout`을 3초로 설정하여 연결 가능 여부만 확인한다.

### Shutdown 시 WARN 로그

테스트 종료 시 아래와 유사한 WARN 로그가 출력될 수 있다:

```
WARN ... Closing org.springframework.boot.web.servlet.context.AnnotationConfigServletWebServerApplicationContext ...
```

이는 테스트 컨텍스트 종료 과정에서 발생하는 정상적인 로그이며, 무시해도 무방하다.

## 테스트 작성 팁

### 1. 순수 단위 테스트 우선

이 프로젝트는 Spring Context를 로딩하지 않는 순수 단위 테스트 위주입니다. 새로운 테스트도 이 패턴을 따르세요:

```java
class NewServiceTest {

    private final NewService service = new NewService(/* 의존성 직접 주입 */);

    @Test
    void methodName_scenario_expectedResult() {
        // given
        // when
        // then
    }
}
```

### 2. 테스트 이름 규칙

`메서드명_시나리오_기대결과` 형식을 따릅니다:

```java
@Test
void sanitize_nullInput_returnsRecipe() { ... }

@Test
void executeSql_invalidQuery_returnsError() { ... }

@Test
void suggestRecipes_noMatches_returnsEmptyList() { ... }
```

### 3. DuckDB는 실제 사용

`DuckDbExecutor`의 테스트는 모킹하지 않고 실제 DuckDB를 사용합니다. 인메모리 동작이므로 빠르고 실제 SQL 동작을 검증할 수 있습니다.

### 4. Virtual Threads와 테스트

Spring Boot의 Virtual Threads 설정은 테스트에 영향을 주지 않습니다. 테스트는 동기적으로 실행되며, ThreadLocal도 테스트 메서드 단위로 격리됩니다.
